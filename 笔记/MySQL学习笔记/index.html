4.7 数据分组

将数据分成多个逻辑组（表数据的子集），以便单独对每个组进行汇总

4.7.1 GROUP BY子句（创建分组）

GROUP BY 指示MySQL按指定列排序分组数据，计算字段会对每组进行一次汇总

规定：

  • GROUP BY 子句可包含任意数目的列，用于嵌套
  • 对于嵌套分组，数据会在最后规定的分组上进行汇总
  • GROUP BY子句中的列为检索列或有效的表达式（不能是聚集函数或别名）
  • 除了聚集计算字段外，SELECT 语句中出现的列要在 GROUP BY 子句中给出
  • 列值为NULL的行会作为一个分组返回

4.7.2 HAVING 子句（过滤分组）

HAVING 子句允许基于分组的过滤

WHERE 子句 与 HAVING 子句的区别：

  • WHERE 没有分组的概念，过滤的是行
  • HAVING 类似于 WHERE ，支持所有 WHERE 操作符，过滤的是分组，可基于分组聚集值进行过滤
  • WHERE 在数据分组前进行过滤，HAVING在数据分组后进行过滤

4.7.3 分组与排序

ORDER BY 和 GROUP BY 的区别：

ORDER BY GROUP BY
排序输出 分组，输出不一定是分组的顺序
后跟任意列（包括非选择列） 必须且只能跟选择列或表达式列
可选 按组计算聚集时，必须使用

五、全文本搜索

全文本搜索功能用于进行高级的数据查询和选择

LIKE 和 REGEXP 在性能、控制精度，智能化等方面存在很大限制，无法满足需要，而全文本搜索能有效解决。

原理：创建指定列中各词的一个索引，从而针对这些词进行搜索，而不必查看每行或分析和处理每个词。

5.5 使用说明

  • MyISAM 数据库引擎支持全文本搜索，而InnoDB不支持
  • 索引全文本数据时，那些3个或以下字符的短词会被忽略
  • MySQL拥有一个内建的非用词列表，在索引全文本数据时总是忽略这些词
  • 如果一个词出现在50%以上的行中，将其作为非用词忽略，不适用布尔文本搜索（50%规则）
  • 忽略词中单引号
  • 一个词要么不出现，要么至少出现在50%的行中，因此，如果表中行数少于三行则全文本搜索不返回结果
  • 全文本搜索不适用于汉语、日语等语言

六、 视图

6.1 理解

视图是虚拟的表，不包含表中任何列或数据，只包含动态检索数据的查询；

本质上是对 SELECT 语句的封装；

视图拥有与真实表基本相同的使用方式；

每次使用视图都要重新执行检索操作，复杂的视图对性能的影响很大。

6.2 用途

(1) 重用 SQL 语句，简化复杂 SQL 操作

(2) 保护数据，视图使用的是表的特定部分而不是整个表，可以给用户授权只访问视图

6.3 规则和限制

  • 必须唯一命名，不能与其它视图或表同名
  • 视图可以嵌套使用
  • 视图可以和表一起使用（如联结）
  • 外层 SELECT 语句的 ORDER BY 会覆盖视图中的 ORDER BY
  • 要具有足够的访问权限才能创建视图
  • 视图不能索引，也不能有关联的触发器或默认值

6.4 基本使用

创建 :CREATE VIEW+AS

删除：DROP VIEW

更新：CREATE OR REPLACE VIEW或先DROP再CREATE

查看创建视图的语句：SHOW CREATE VIEW

七、存储过程

7.1 理解

存储过程是为以后的使用而保存的一条或多条MySQL语句的集合，与批文件类似

当一个操作涉及到针对多个表的根据条件执行的多条MySQL语句时可创建一个存储过程保存该操作

7.2 优点

  • 将处理封装到易使用的单元中，可简化复杂的操作
  • 使用同一存储过程，不用反复编写，可减少错误，有利于保证数据的一致性
  • 当表名等有变化只需更改存储过程的代码，可简化对变动的管理，有利于提高安全性
  • 使用存储过程比使用单独的SQL语句更快，有利于提高性能
  • 利用只能在单个请求中使用的MySQL元素和特性，可功能更强编写更灵活的代码
  • 编写和执行存储过程的权限是分开的，即使不能编写也能使用存储过程

八、触发器

8.1 理解

用于在某个表发生更改时自动执行某些语句完成所需功能

触发器是响应 INSERT、DELETE、UPDATE 语句而自动执行的一条MySQL语句(或BEGIN和END之间的一组语句)

除了以上三种MySQL语句其它不支持触发器

只有表支持触发器，视图或临时表不支持

触发器按每个表每个事件每次地定义，每个表每个事件每次只允许一个触发器。每个表最多支持6个触发器（每条 INSERT 、 UPDATE 和 DELETE 的之前和之后）

如果 BEFORE 触发器失败，则 MySQL 将不执行请
求的操作。如果 BEFORE 触发器或语句本身失败，MySQL将不执行 AFTER 触发器。

8.2 基本使用

8.2.1 创建触发器

给出：

  • 唯一的触发器名
  • 触发器关联的表
  • 触发器应该响应的活动（ DELETE 、 INSERT 或 UPDATE ）
  • 触发器何时执行（BEFORE或AFTER）

8.2.2 使用触发器

INSERT 触发器

  • 在 INSERT 触发器代码内，可引用一个名为 NEW 的虚拟表，访问被插入的行；
  • 在 BEFORE INSERT 触发器中， NEW 中的值也可以被更新（允许更改被插入的值）；
  • 对于 AUTO_INCREMENT 列， NEW 在 INSERT 执行之前包含 0 ，在 INSERT执行之后包含新的自动生成值

DELECT 触发器

  • 在 DELETE 触发器代码内，可引用一个名为 OLD 的虚拟表，访问被删除的行；
  • OLD 中的值全都是只读的，不能更新。

UPDATE触发器

  • 可以引用名为 OLD 的虚拟表访问以前的值，引用名为 NEW 的虚拟表访问新更新的值；
  • NEW 中的值可能也被更新（允许更改将要用于 UPDATE 语句中的值）；
  • OLD 中的值全都是只读的，不能更新。

8.2.3 删除触发器

8.2.4 注意事项

1.对于 INSERT 和 UPDATE 触发器，BEFORE 常用于数据验证和净化，有利于保障数据的一致性

2.创建触发器需要特殊的安全访问权限，但执行是自动的；

3.可用触发器来创建审计跟踪，记录数据的变化

九、事务机制

9.1 理解

事务处理机制用来管理必须成批执行的MySQL操作，这组操作要么全部执行要么完全不执行，用于维护数据库的完整性；

如果没有错误发生，整组语句提交给数据库表。如果发生错误，则进行回退以恢复数据库到某个已知且安全的状态。

常用术语：

事务（ transaction ）：一组SQL语句；
回退（ rollback ）：撤销指定SQL语句的过程；
提交（ commit ）：将未存储的SQL语句结果写入数据库表；
保留点（ savepoint ）：事务处理中设置的临时占位符，可对它发布回退（与回退整个事务不同）

9.2 控制事务处理

将SQL语句组分解为逻辑块，对何时写数据，何时撤销进行明确的管理。

注意：

InnoDB 引擎支持事务处理功能而 MyISAM 不支持；

事务开始的标识为START TRANSACTION，回退命令为ROLLBACK，提交命令为COMMIT

当 COMMIT 或 ROLLBACK 语句执行后，事务会自
动关闭

9.2.1 ROLLBACK

ROLLBACK 命令用来回退（撤销）MySQL语句

只能在一个事务处理内使用

可以回退 INSERT 、 UPDATE 和 DELETE 语句，但SELECT 语句、 CREATE 或 DROP 操作不能回退

9.2.2 COMMIT

在事务处理块中，提交不会隐含进行，使用 COMMIT 语句明确地提交

9.2.3 SAVEPOINT

SAVEPOINT 语句在事务处理块中合适的位置放置占位符(保留点)，用于回退部分事务处理

9.2.4 默认提交行为

一般MySQL语句都是直接针对数据库表执行且立即生效，即提交操作是自动进行的，称为隐含提交

可使用SET autocommit = 0更改默认的自动提交行为，此时COMIMIT也不会生效，直至再次将其设为 1。

该设置针对的是每个连接不是服务器

  • 校对：规定字符如何比较的指令
